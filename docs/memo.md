## 対応判断（忖度なし）

### 1. 空タブエリアを閉じる
**判断: やる（優先度・低）→ 対応済み（チャットタブ）**

- 一文目は望ましい挙動、二文目は現状の問題（空のタブ帯が残ったまま開いている）の指摘。同じ要件の裏表。
- エディタ `TabBar` は既に `openFiles.length === 0` で非表示。問題の本体はチャット側で、最終タブを閉じると空セッションを自動生成していたこと。
- **やり方（実装）:** 最終オープンチャットを閉じたら空タブを作らず `showChat: false`。パネル再表示・新規作成・履歴再開・コンポーザ挿入時に必要なら空セッションを用意。

### 2. 未保存のままアプリ終了で消える
**判断: やる（優先度・最高）→ 対応済み**

- SPEC は autosave off・明示保存のみ。その前提なら終了時ガードがないのは欠陥。
- フォルダ閉じには dirty 確認があるのにアプリ終了に無いなら、ユーザーは「保存したつもり」で落ちる。
- **やり方（実装）:** ウィンドウ `close` / メニュー終了をインターセプト。dirty があればネイティブダイアログ（保存して終了 / 保存しない / キャンセル）。オートセーブは入れない。

### 3. エクスプローラで F2 リネーム
**判断: やる（優先度・中）→ 対応済み**

- コンテキストメニューのリネームは既にある。Delete ショートカットもあるのに F2 だけ無いのは不揃い。
- Windows エクスプローラ／VS Code の期待値。実装も既存 inline rename にキーを足す程度。
- **やり方（実装）:** 選択1件・ルート／プレビュー以外で F2 → 既存 `startRename`。複数選択・メニュー表示・inline 入力・削除確認中は無視。

### 4. 新規ファイルを Markdown にする
**判断: デフォルト固定はしない → 空の Markdown テンプレで対応済み**

- Compass は「テキスト全般」の AI workspace。code / data 用途もある。`.md` 固定は押し付け。
- 「空のファイル」は `.txt` のまま。MD が欲しければ「新規… → テンプレートから… → 空の Markdown」（`untitled.md`、本文空）。
- **やり方（実装）:** 内蔵テンプレ `blank-markdown` をメニュー先頭に追加。設定やプリセット連動はしない。

### 5. 新規作成時に拡張子以外を選択
**判断: やる（優先度・高〜中）→ 対応済み**

- 名前だけ打ち直して Enter、が自然。拡張子ごと選択だと `.txt` を消しがち。
- Explorer / VS Code と同じ。
- **やり方（実装）:** `getNameStemSelectionEnd` で初期選択。ファイルは最後の `.` 直前まで、フォルダ／ドット無し／`.gitignore` 等は全選択。新規作成とリネームの両方に適用。

### 6. Word / Excel を外部アプリで開く
**判断: やる（優先度・中）→ 対応済み（ビューアは作らない）**

- Office バイナリを Monaco に食わせるのは無意味で危険（文字化け・巨大・破損リスク）。
- 製品の中核はテキスト編集＋AI。Office 編集はスコープ外。
- **やり方（実装）:** `.doc(x)` / `.xls(x)` / `.ppt(x)` / ODF 等はダブルクリックで `shell.openPath`。コンテキストメニューに「既定のアプリで開く」を常設（任意ファイル）。CSV はテキストのまま。

## 対応判断（忖度なし）続き

### 7. エディタ領域のタブの一括操作が必要では？
**判断: エディタ限定で Close / Close Others / Close All を入れるのは妥当。必須ではないが、やるなら今が安い。→ 対応済み**

- **スコープ:** エディタだけ、で正しい。チャット／ターミナルまで揃える理由はない（項目 8）。
- **3項目だけ、も正しい。** Close to the Right / Close Saved は Compass のタブ量では過剰。Close は × と重複するが、右クリックの定番セットとして残してよい（単独省略するほどの害もない）。
- dirty 確認（項目 9）が済んだので、以前より条件は良い。`prepareCloseFiles` に対象タブ配列を渡すだけで Others / All の安全側が揃う。
- それでも欠陥ではない。× 連打で足りる。優先度はデータロス系より下。
- 効く場面は明確: AI 探索やプレビューでタブが増えたあと **Close Others**。Close All はフォルダ切り替え前の掃除程度。
- **やり方（実装）:** `TabBar` 右クリックメニュー。Close / Close Others / Close All。閉じる前に `prepareCloseFiles`。タブ1枚のとき Others は無効。右クリックで当該タブをアクティブ化。

### 8. エディタ・AIチャット・ターミナルのタブ操作を統一すべき？
**判断: 3領域の統一プロジェクトはしない。チャットには Close / Close Others / Close All を入れる（対応済み）。ターミナルは入れない。**

- 「タブに見える・切替・×で閉じる・並べ替え」は既にエディタ／チャットで近い。ターミナルは並べ替えすら無いが、タブ数が少ないので致命傷ではない。
- 意味が違うものを同じ操作セットに押し込むのは間違い:
  - **エディタ:** ファイル・dirty・プレビュー／ブラウザ／設定タブ。閉じる＝編集バッファ破棄の可能性。
  - **チャット:** 閉じても履歴は残る。一括閉じの事故コストは低い。必須ではないが害もほぼ無いのでエディタと同じ3項目を入れてよい。
  - **ターミナル:** 閉じる＝PTY kill。誤って一括閉じると作業が消える。ここには揃えない。
- 共有してよいのは `tab-reorder` のような**プリミティブ**と、見た目の一貫性。ドメインロジックまで共通コンポーネント化して操作を揃えるのは、統一のための統一。
- **やり方:** 統一プロジェクトは立ち上げない。チャットは `ChatPanel` 右クリックでエディタと同じ Close / Close Others / Close All（履歴は残る）。ターミナルは現状維持。

### 9. エディタタブを閉じると未保存変更が黙って消える
**判断: 欠陥。直す（優先度・高）。一括操作より先。→ 対応済み**

- 現状: `closeFile` は dirty を見ない。● が付いていても × 一発で破棄。アプリ終了だけ保存／破棄／キャンセルがある。
- SPEC は autosave off・明示保存のみ。その前提でタブ閉じが無確認なのは、終了ガードを穴あけしているのと同じ。日常で失う経路は「終了」より「タブを閉じる」のほうが多い。
- ● は「まだディスクに無い」という約束。確認なしで消すならマークの意味が嘘になる。
- フォルダ閉じ・エンコーディング変更・終了には確認があるのにタブ × だけ無いのは不揃いであり、意図したデザインとは思えない。
- オートセーブで逃げるのは不可。製品方針と衝突する。
- **やり方（実装）:** dirty かつ保存対象（`listDirtySavableFiles`）なら、閉じる前にネイティブダイアログ（保存 / 保存しない / キャンセル）。`prepareCloseFiles` を TabBar の × から呼ぶ。クリーン・プレビュー・非テキストは即閉じ。保存失敗時は閉じない。
